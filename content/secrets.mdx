# Dashboard Secrets Setup

Secrets are managed differently depending on the environment:

- **Local development** вЂ” [Doppler](https://www.doppler.com/) CLI injects secrets at runtime
- **CI/CD & production** вЂ” GitHub Actions secrets, scoped to the `production` environment, push values to Render at deploy time

---

## Local Development (Doppler)

Local development no longer uses `.env` files. Doppler acts as a secrets manager that injects environment variables into the process at startup.

### First-time setup

```bash
# Authenticate with Doppler
doppler login

# Link the local directory to the correct Doppler project/config
doppler setup
```

Select the appropriate project and `dev` config when prompted.

### Running services

Prefix your usual start command with `doppler run --`:

```bash
# Frontend
doppler run -- npm run dev

# Spring services
doppler run -- mvn spring-boot:run
```

All environment variables are injected automatically вЂ” no `.env` files needed.

---

## CI/CD (GitHub Actions)

Both Spring services have a GitHub Actions pipeline at `.github/workflows/ci.yml`. Secrets are stored in the **GitHub repository secrets**, scoped to the `production` environment.

### Pipeline stages

```
push to master
  в””в”Ђв–є build в†’ unit-test в”¬в”Ђв–є deploy (production env only)
                        в””в”Ђв–є integration-test
```

The `deploy` job does **not** build a Docker image вЂ” it calls the Render API to push updated environment variable values to the running service.

### How secrets reach Render

```bash
curl -X PUT "https://api.render.com/v1/services/${RENDER_SERVICE_ID}/env-vars" \
  -H "Authorization: Bearer ${RENDER_API_KEY}" \
  -d '[{"key": "JWT_SECRET", "value": "..."}]'
```

This means secrets are always sourced from GitHub and pushed at deploy time. Render never holds stale values from a previous deploy.

## Secrets Reference

### `dashboard-spring-oauth`

All of the following must be set as GitHub repository secrets in the `production` environment.

| Secret | Description |
|--------|-------------|
| `RENDER_SERVICE_ID` | Render service identifier for the OAuth server |
| `RENDER_API_KEY` | Render API key used to push env vars |
| `SPRING_DATA_MONGODB_URI` | Full MongoDB connection string (via `MONGO_USER`, `MONGO_PWD`, `MONGO_HOST`, `MONGO_DB`) |
| `JWT_SECRET` | HS256 signing key (min 256 bits) |
| `JWT_EXPIRATION` | Access token lifetime in ms |
| `JWT_REFRESH_EXPIRATION` | Refresh token lifetime in ms |
| `OIDC_ISSUER` | Public OIDC issuer URL |
| `SPRING_SECURITY_OAUTH2_SECRET` | Shared secret for OAuth2 client authentication |
| `GRAFANA_API_KEY` | Grafana Cloud API token for Loki logging |
| `GRAFANA_URL` | Loki push endpoint |
| `R2_ACCESS_KEY_ID` | Cloudflare R2 S3-compatible access key |
| `R2_SECRET_ACCESS_KEY` | Cloudflare R2 secret key |
| `R2_ACCOUNT_ID` | Cloudflare account ID |
| `R2_BUCKET_NAME` | R2 bucket name |
| `RESEND_APIKEY` | Resend email service API key |
| `EMAIL_BASEURL` | Frontend base URL used in verification emails |
| `EMAIL_FROMADDRESS` | Sender address for outgoing emails |
| `EMAIL_VERIFICATIONTOKENEXPIRATIONMS` | Email verification token lifetime in ms |

### `dashboard-spring-data`

| Secret | Description |
|--------|-------------|
| `RENDER_SERVICE_ID` | Render service identifier for the data API |
| `RENDER_API_KEY` | Render API key used to push env vars |
| `GRAFANA_API_KEY` | Grafana Cloud API token for Loki logging |
| `JWT_SECRET` | Must match the value in `dashboard-spring-oauth` |

### `dashboard-frontend`

The frontend is not deployed via GitHub Actions. The following secrets must exist in Doppler (dev config) and in whatever deployment environment the frontend runs in.

| Variable | Description |
|----------|-------------|
| `AUTH_SECRET` | NextAuth session encryption key |
| `OAUTH2_CLIENT_SECRET` | Client secret registered on the OAuth server |
| `NEXT_PUBLIC_LOKI_TOKEN` | Grafana token for client-side log shipping |

Non-secret configuration (API URLs, versions, etc.) lives in `.env` committed to the repository.

## Adding a new secret

1. Add the value to Doppler under the relevant project/config (for local dev).
2. Add the same value to GitHub repository secrets under **Settings в†’ Secrets в†’ Actions в†’ `production` environment**.
3. If the secret needs to reach Render at runtime, add it to the `curl` payload in the `deploy` job of the relevant `.github/workflows/ci.yml`.
4. Reference it in the Spring `application.properties` via `${ENV_VAR_NAME:default}` or in Next.js via `process.env.VAR_NAME`.
